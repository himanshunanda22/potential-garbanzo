Chapter 00

Where this started — DeceptNet v1

Before understanding why we needed MDP and Q-learning, you need to understand what we built first, how it worked, and exactly where it broke down. That story is the entire motivation for everything that follows.

The original idea

The core concept was simple and elegant: instead of blocking attackers — which alerts them and drives them to adapt — deceive them. Let them think they succeeded. Return convincing fake data. Meanwhile, capture their full fingerprint silently and let them exhaust their playbook against a system that was never real.

Traditional intrusion detection systems answer one question: is this malicious? If yes, block. This has two problems. Blocking reveals detection — the attacker now knows a defence exists and changes approach. And it discards the attacker the moment they're identified, throwing away every future request that could teach us about their tools, targets, and technique.

The deception principle: An attacker who believes they are succeeding will keep probing. Every request they send against a fake system is a request that reveals something about their methodology, their tooling, their next target. The longer we can maintain the illusion, the richer the intelligence we collect — and the more we can train our models on real adversarial behaviour.

The decision is made entirely from the current request x_t, processed independently of every other request. There is no memory. There is no session context. p̂ is computed, thresholded, and the request is routed. That's the complete decision logic of v1.

What v1 got right

v1 was not naive. The deception principle is genuinely sound. The 13-feature extraction covers the most important attack signatures. ThreatNet achieves ~92% validation accuracy on held-out synthetic data. The fingerprinting pipeline captures useful intelligence. For a first version, this is a solid foundation.

Where v1 breaks — the three hard limitations

Despite the solid foundation, v1 has three fundamental limitations. Each one is not an implementation bug — it is a structural consequence of the stateless binary design. No amount of tuning fixes them without a different architecture.

Limitation 1 — No memory across requests
v1 sees each request in complete isolation. An attacker can send four clearly escalating requests — recon, probe, injection, repeat — and v1 treats request 5 with exactly the same logic as request 1. The session context that distinguishes a persistent sophisticated attacker from a one-off probe is entirely invisible.
v1: action(x₅) depends only on x₅
v2: action(x₅) depends on [x₁, x₂, x₃, x₄, x₅] ← the whole story
Limitation 2 — Binary action with no strategy
v1 has exactly two possible actions: deceive or pass through. This misses the entire strategic dimension of deception. Sometimes the right move is to return richer fake data to keep an attacker probing longer (DEEPEN_TRAP). Sometimes it's to trigger an alert while maintaining the facade (ESCALATE). Sometimes the attacker has seen through the deception and the right move is an abrupt connection drop (HARD_EXIT). v1 can't make any of these distinctions.
Limitation 3 — No learning from outcomes
After v1 responds to a request, it discards the interaction entirely. It doesn't ask: was that the right call? Should I have waited longer before escalating? Did the attacker send more useful requests after I deceived them, suggesting I should have used a richer fake response? v1 has no feedback loop — it cannot improve from experience. The same decision rule that made its first decision makes its millionth decision.
The fundamental gap: v1 answers the question "is this request bad?" on a per-request basis. But the real question is "given this attacker's entire session history so far, what action maximises the total intelligence I extract from them over the rest of their session?" That is a sequential decision problem under uncertainty with a long-run objective. It is precisely what Markov Decision Processes and reinforcement learning were designed to solve.

The upgrade path — what v2 adds on top of v1

Crucially, v2 does not discard v1. It keeps everything that worked — the ThreatNet classifier, the 13-feature extraction, the deception engine, the fingerprint capture, the SQLite store — and adds the MDP decision layer on top. v1's ThreatNet still runs on every request. Its output (threat score + type) feeds into the MDP state. The classifier becomes an input feature to the decision engine, not the decision engine itself.

v1 pipeline:  request → φ(x) → ThreatNet → {deceive | pass}
v2 pipeline:  request → φ(x) → ThreatNet → session state s_t → Q-network → argmax Q(s,a) → {DECEIVE | DEEPEN | ESCALATE | EXIT}

The MDP layer adds memory (session state encoding 8 past requests), strategy (four actions with different risk/reward profiles), and learning (online Bellman updates after every real attacker interaction). The rest of the document explains exactly why each of these additions works the way it does — starting with why memory alone isn't enough, and why we need the infinite horizon.

Chapter 01

Why isn't a classifier enough?

Before we get to the maths, we need to understand the problem that made all of it necessary.

The original DeceptNet v1 used a neural network to answer one question per request: "is this request malicious?" The answer was a single number — the threat probability \(\hat{p} \in [0,1]\). If \(\hat{p} > 0.65\), return a fake response. Otherwise pass through.

This works. But it misses something fundamental about how attackers behave.

The key observation: Attackers don't send one request. They send a session — a sequence of requests that reveals intent, skill level, and methodology over time. A single request seen in isolation is much less informative than that same request seen as the 5th in a probing sequence.

Consider two scenarios where the same request arrives — a POST with a SQL-like payload:

Scenario Av1 sees the same thing
First request from this IP. Could be a security researcher, an automated scanner, or an accidental misconfiguration. Threat score: 0.74. v1 returns a fake response.

Best action: DECEIVE — keep watching.
Scenario Bv1 sees the same thing
Fifth request from this IP. The previous four escalated from reconnaissance → SQL probe → UNION attack → identical repeat. This is a sophisticated, persistent attacker.

Best action: ESCALATE — we have enough evidence.

v1 gives the same answer in both cases because it only sees the current request. v2 needs to reason over the entire session history to make the right call. That is precisely what a Markov Decision Process formalises.

The decision problem: At each timestep \(t\), the system observes a state \(s_t\) (the full session history encoded as a vector), and must choose an action \(a_t\) from {DECEIVE, DEEPEN_TRAP, ESCALATE, HARD_EXIT}. The goal is to maximise the total reward accumulated over the entire session — not just the reward from the current step.

This is fundamentally different from classification. A classifier asks: what is true right now? An MDP agent asks: what should I do, given everything I've seen so far, to maximise what I gain from this entire interaction?

Chapter 02

The finite-to-infinite projection

This is the conceptual heart of the system. What does it mean to "project a finite trajectory into an infinite horizon"? Let's unpack it carefully.

The finite observation problem

Every attacker session has a finite length. An attacker sends 3, 5, maybe 10 requests. At step 4, we have observed 4 feature vectors: \(\phi_1, \phi_2, \phi_3, \phi_4\). This is our finite trajectory.

The problem: we want to make a decision whose consequences extend beyond the current step. If we ESCALATE now, we stop the session. If we DECEIVE, we get more data. The value of an action depends not just on what happens now, but on what could happen across all future steps — and in principle, that horizon is unbounded.

The state encoding — finite trajectory → fixed-size vector
\[s_t = \bigl[\underbrace{\phi_{t-W+1}, \phi_{t-W+2}, \ldots, \phi_t}_{W \text{ feature vectors}}\bigr] \in \mathbb{R}^{W \cdot d}\]
We take the last \(W = 8\) requests (padding with zero vectors if the session is shorter) and concatenate their feature vectors. With \(d = 13\) features per request, this gives a fixed 104-dimensional state vector regardless of how many requests have been seen. This is the finite part — a snapshot of the last 8 steps.
\(W = 8\)Window size — we remember the last 8 requests \(d = 13\)Feature dimension per request (SQL score, XSS score, etc.) \(s_t \in \mathbb{R}^{104}\)State vector — fixed size regardless of session length

Good — we now have a fixed-size vector encoding recent history. But we still haven't solved the infinite horizon problem. The state is finite. How do we reason about infinite futures?

The discount factor γ — the mathematical projection

Here is the key idea. Instead of trying to enumerate all possible futures (impossible), we define the value of being in a state as a convergent infinite sum, weighted by powers of a discount factor \(\gamma \in [0,1)\):

The infinite-horizon return — the sum that never has to be computed directly
\[G_t = \sum_{k=0}^{\infty} \gamma^k r_{t+k} = r_t + \gamma r_{t+1} + \gamma^2 r_{t+2} + \gamma^3 r_{t+3} + \cdots\]
This sum extends to infinity — it accounts for rewards at every future step. But notice: each term is multiplied by \(\gamma^k\). Since \(\gamma = 0.95 < 1\), the terms shrink geometrically. The sum converges — it has a finite value even though it sums infinitely many terms. This is how a finite computation represents an infinite future.
\(G_t\)The total discounted return from time t — what we actually want to maximise \(\gamma = 0.95\)Discount factor — a reward 10 steps away is worth \(0.95^{10} \approx 0.60\) today \(r_{t+k}\)The reward received \(k\) steps in the future
Why this works as a projection: The discount factor \(\gamma\) doesn't truncate the future — it compresses it. A reward 100 steps away contributes \(0.95^{100} \approx 0.006\) — nearly zero, but not exactly zero. The infinite sum has a well-defined finite value. So by learning \(Q(s, a) \approx \mathbb{E}[G_t]\), we are encoding information about every possible future in a single number derived from a 104-dimensional input. That is the projection: infinite time horizon → finite-dimensional function.

Let's make this concrete with an interactive demo. Adjust \(\gamma\) and see how future rewards are weighted:

Notice what happens at extreme values: \(\gamma \to 0\) means the agent only cares about the immediate reward — pure myopia, no planning. \(\gamma \to 1\) means all future rewards are weighted equally — the sum may not converge, and the agent cannot distinguish near from far consequences. The sweet spot \(\gamma = 0.95\) gives a "planning horizon" of roughly \(\frac{1}{1-\gamma} = 20\) effective steps.

Effective planning horizon
\[\text{Effective horizon} \approx \frac{1}{1-\gamma} = \frac{1}{1-0.95} = 20 \text{ steps}\]
This is a rule of thumb: beyond \(\frac{1}{1-\gamma}\) steps, the discounted weights become negligible. With \(\gamma = 0.95\), the system effectively reasons about the next ~20 attacker requests — far beyond any real session, ensuring true infinite-horizon planning in practice.

Why this matters for DeceptNet

An attacker session typically has 3–10 requests. Our effective horizon is 20. This means the Q-function, trained over thousands of sessions, learns to encode: "if I DECEIVE here, I expect the attacker to send 2 more requests, and then I can ESCALATE with high confidence, yielding total reward X." That forward-looking reasoning is the projection — it lives inside the Q-values, invisible but always present.

Chapter 03

The Bellman equation — why it's the key

The Bellman equation is not just a formula. It is a recursive definition of value that makes the entire enterprise of computing \(G_t\) tractable. Understanding it deeply is understanding RL itself.

The Q-function

Define the action-value function (the Q-function) as the expected discounted return when taking action \(a\) in state \(s\), then acting optimally forever after:

The Q-function — the central object of Q-learning
\[Q^\pi(s, a) = \mathbb{E}_\pi\!\left[\sum_{k=0}^{\infty} \gamma^k r_{t+k} \;\middle|\; s_t = s,\; a_t = a\right]\]
Q(s,a) is a single number that encodes the answer to: "if I'm in state s, take action a right now, and then follow policy π forever — what total discounted reward do I expect?" We want the optimal version Q*, where π is the best possible policy.

Computing this directly would require simulating every possible future — infinite computation. The Bellman equation provides a recursive shortcut.

The recursive decomposition

Notice that the infinite sum \(G_t = r_t + \gamma r_{t+1} + \gamma^2 r_{t+2} + \cdots\) can be rewritten as:

The key algebraic manipulation
\[G_t = r_t + \gamma \underbrace{(r_{t+1} + \gamma r_{t+2} + \gamma^2 r_{t+3} + \cdots)}_{= G_{t+1}}\]
\[\therefore \quad G_t = r_t + \gamma G_{t+1}\]
The return from step t equals the immediate reward plus γ times the return from step t+1. This is the recursive structure. The future has already been summarised in G_{t+1} — we don't need to expand it further.

Taking expectations and substituting the Q-function, we get the Bellman optimality equation:

Bellman optimality equation — the equation that defines Q*
\[Q^*(s, a) = \mathbb{E}\!\left[\,r + \gamma \max_{a'} Q^*(s', a') \;\middle|\; s, a\right]\]
The optimal Q-value of taking action a in state s equals the expected immediate reward r, plus γ times the value of the best action in the next state s'. Notice Q* appears on both sides — this is a self-consistency condition. Any function satisfying this equation IS the optimal Q-function.
\(r\)Immediate reward from the reward shaping function R(s, a) \(s'\)Next state — the new 104-dim session vector after appending the next request \(\max_{a'} Q^*(s',a')\)The value of the best action in the next state — this is the "future" compressed into one number
Why this is profound: The Bellman equation converts an infinite-horizon problem into a one-step lookahead problem. Instead of summing rewards to infinity, we only need to look one step ahead and then ask: "what's the best Q-value in the next state?" If we knew Q* already, we could compute Q*(s,a) from just one step of lookahead. The trick is: we don't know Q* yet. So we use this self-consistency condition to iteratively improve our estimate until it converges.

The TD error — how we learn from the equation

We can't solve the Bellman equation analytically — the state space is continuous and 104-dimensional. Instead, we use the equation to define an error signal (the Temporal Difference error) and gradient descent to eliminate it:

Temporal Difference (TD) error — the learning signal
\[\delta_t = \underbrace{r_t + \gamma \max_{a'} Q(s_{t+1}, a')}_{\text{TD target }y_t} - \underbrace{Q(s_t, a_t)}_{\text{current prediction}}\]
The TD error measures how wrong our current Q estimate is relative to the Bellman target. If δ > 0, the target is higher than we predicted — we underestimated this action's value. If δ < 0, we overestimated it. We use gradient descent to push our Q estimate toward the target, shrinking δ over time.

The key insight from experimenting: α (learning rate) controls how quickly we trust new information. Too high and the Q values oscillate. Chapter 04

Why a neural network? What does DQN add?

Q-learning was invented in 1989. Why do we need a deep neural network? What does DQN actually do differently from classic Q-learning?

The curse of dimensionality

Classic Q-learning maintains a lookup table: one Q-value per (state, action) pair. For our system, the state is a 104-dimensional continuous vector. The number of possible states is effectively infinite — you cannot build a table for them.

The dimensionality problem: Our state space \(\mathcal{S} = \mathbb{R}^{104}\). Even if we discretised each dimension into just 10 bins, the table would need \(10^{104}\) entries — more than the number of atoms in the observable universe. Classic Q-learning is fundamentally incompatible with continuous, high-dimensional state spaces.

The solution: instead of a table, use a function approximator — a neural network \(Q_\theta(s, a)\) parameterised by weights \(\theta\). The network takes the 104-dim state as input and outputs Q-values for all 4 actions simultaneously. We train it with gradient descent using the TD error as the loss signal.

DQN — replacing the table with a neural network
\[Q_\theta(s, a) \approx Q^*(s, a)\]
\[\theta \leftarrow \theta - \alpha \nabla_\theta \mathcal{L}(\theta)\]
\[\mathcal{L}(\theta) = \mathbb{E}\!\left[\,\underbrace{\bigl(y_t - Q_\theta(s_t, a_t)\bigr)^2}_{\text{squared TD error}}\,\right]\]
The network learns to approximate Q* by minimising the squared TD error over minibatches of past transitions sampled from the replay buffer. The gradient ∇θL tells us how to adjust every weight in the network to make Q_θ closer to the Bellman target y_t.

The architecture of our Q-network

Our Q-network (called ThreatNet-Q) takes the 104-dim state vector and outputs 4 Q-values — one per action:

Forward pass — 104 → 256 → 128 → 64 → 4
\[h_1 = \text{ReLU}(\text{LayerNorm}(W_1 s + b_1)) \in \mathbb{R}^{256}\] \[h_2 = \text{ReLU}(\text{Dropout}(W_2 h_1 + b_2)) \in \mathbb{R}^{128}\] \[h_3 = \text{ReLU}(W_3 h_2 + b_3) \in \mathbb{R}^{64}\] \[\mathbf{q} = W_4 h_3 + b_4 \in \mathbb{R}^4\]
No softmax or sigmoid on the output — Q-values can be any real number (positive or negative). LayerNorm on the first layer stabilises training with the wide range of input magnitudes. Dropout prevents overfitting. The output vector q has one entry per action: q[0]=Q(s,DECEIVE), q[1]=Q(s,DEEPEN_TRAP), etc.

The experience replay buffer

There's a subtle but critical problem with naïve DQN: if we update the network after every single request, consecutive transitions are highly correlated — they come from the same session, the same attacker, with similar states. Training on correlated data breaks the i.i.d. assumption gradient descent needs and causes catastrophic forgetting.

The fix is the experience replay buffer: we store every transition \((s_t, a_t, r_t, s_{t+1}, \text{done})\) in a circular buffer of size 10,000. At each training step, we sample a random minibatch of 64 transitions — breaking the correlations.

Why random sampling works: By drawing uniformly from the entire buffer, each gradient step mixes transitions from many different sessions, different attackers, different threat types. The gradient estimate is now approximately i.i.d. — each mini-batch is a representative sample of the agent's experience, not a slice of one correlated trajectory.
Chapter 05

The Double DQN fix — and why it matters

Vanilla DQN has a systematic bias that causes it to overestimate Q-values. Double DQN fixes it with an elegant two-network trick. This is not a minor tweak — it's the difference between a policy that works and one that doesn't.

The overestimation problem

In vanilla DQN, the TD target is:

Vanilla DQN target — the biased version
\[y_t^{\text{DQN}} = r_t + \gamma \max_{a'} Q_\theta(s_{t+1}, a')\]
Notice: the same network θ is used for both selecting the best next action (the argmax) AND evaluating that action's Q-value. This is the problem.

Why is using the same network for both selection and evaluation a problem? Consider this: the network's Q-estimates have noise — random errors from imperfect training. When we take \(\max_{a'} Q_\theta(s', a')\), we are systematically picking the action whose Q-value has the most positive noise. Then we evaluate that same action with the same noisy network. The result: a consistently overestimated target.

The max operator and noise: Suppose the true Q-values are all equal at 3.0, but our estimates are noisy: [3.2, 2.8, 3.1, 2.9]. The max picks 3.2 — already overestimated. We then use this inflated value as our target. Over thousands of updates, this causes systematic upward drift. The agent becomes overconfident about action values and learns a suboptimal policy.

The Double DQN solution — decouple selection from evaluation

The fix is to use two separate networks for the two roles. The online network \(Q_\theta\) selects which action is best. The target network \(Q_{\theta^-}\) evaluates that action's value.

Double DQN target — the unbiased version
\[y_t^{\text{DDQN}} = r_t + \gamma\; Q_{\theta^-}\!\!\left(s_{t+1},\; \underbrace{\arg\max_{a'} Q_\theta(s_{t+1}, a')}_{\text{online network selects action}}\right)\]
Step 1: use the online network θ to find which action looks best: a* = argmax Q_θ(s', a'). Step 2: use the target network θ⁻ to evaluate that specific action: Q_θ⁻(s', a*). The two networks have different errors — using them for different roles eliminates the systematic bias.
\(Q_\theta\)Online network — trained every step, always up to date, used for action selection \(Q_{\theta^-}\)Target network — copy of θ from 50 steps ago, used for evaluation only, held frozen between copies

The target network \(Q_{\theta^-}\) is periodically synced to the online network:

Target network synchronisation
\[\theta^- \leftarrow \theta \quad \text{every } N_{\text{sync}} = 50 \text{ steps}\]
Every 50 gradient steps, we copy the online network's weights to the target network. Between copies, the target network is frozen. This makes the TD target stable — we're not chasing a moving target. If we updated θ⁻ every step, the targets would shift as fast as the network, causing oscillation or divergence.

The Huber loss — robust to outlier transitions

One more detail: instead of squared error (MSE) for the loss, we use Huber loss (also called Smooth L1). Why? Occasionally a transition has an extremely large TD error — an outlier. MSE amplifies outliers quadratically. Huber loss behaves like L2 for small errors but L1 for large ones, giving bounded gradients.

Huber loss — robust loss function
\[\mathcal{L}_\delta = \begin{cases} \frac{1}{2}\delta^2 & \text{if } |\delta| \leq 1 \\ |\delta| - \frac{1}{2} & \text{if } |\delta| > 1 \end{cases}\]
For small errors (|δ| ≤ 1): smooth quadratic, same as MSE — gentle updates near the optimum. For large errors (|δ| > 1): linear, same as MAE — bounded gradient magnitude even for huge outlier transitions. This prevents a single bad transition from causing a catastrophically large weight update.
Chapter 06

The full picture — everything connected

Now we can see how every component fits together into one coherent system. Each piece solves a specific problem. Together, they form a system that does something none of them could do alone.
1

Feature extraction solves the representation problem

Raw HTTP requests are unstructured text. The feature extractor \(\phi(x)\) converts them into a 13-dimensional vector of meaningful signals. Each dimension is normalised to [0,1], making the state space well-behaved for the neural network.

φ: HTTP request → ℝ¹³ (SQL score, XSS score, scanner score, ...)
2

Session windowing solves the temporal context problem

A single feature vector is stateless. Concatenating the last W=8 vectors gives the agent memory — it can reason about sequences, not just individual requests. Zero-padding handles sessions shorter than W.

s_t = [φ_{t-7}, ..., φ_t] ∈ ℝ¹⁰⁴ (the finite encoding)
3

The discount factor solves the infinite horizon problem

We can't enumerate infinite futures. The discount factor \(\gamma = 0.95\) converts the infinite-horizon return into a convergent geometric series. The Q-function encodes this infinite sum implicitly — one trained number per (state, action) pair.

G_t = r_t + γr_{t+1} + γ²r_{t+2} + ... = finite number
4

The Bellman equation solves the computation problem

We can't compute \(G_t\) by summing to infinity. The Bellman equation converts the infinite sum into a one-step recursion: Q*(s,a) = r + γ·max Q*(s'). We only need to look one step ahead.

y_t = r + γ · max_a' Q_θ⁻(s', a') ← one-step target
5

The neural network solves the continuous state problem

We can't store a Q-table for a 104-dimensional continuous space. The Q-network \(Q_\theta\) is a universal function approximator — it generalises across similar states it has never seen before.

Q_θ: ℝ¹⁰⁴ → ℝ⁴ (104-dim state → 4 Q-values)
6

The replay buffer solves the correlation problem

Consecutive transitions from the same session are highly correlated. Training on correlated data breaks gradient descent. The buffer stores 10,000 transitions and random sampling breaks the correlations.

|D| = 10,000 transitions → random minibatch of 64 per update
7

Double DQN solves the overestimation problem

Vanilla DQN uses the same network to select and evaluate actions, causing systematic overestimation. Two networks — online (selects) and target (evaluates) — decouple these roles, eliminating the bias.

a* = argmax Q_θ(s') → Q_θ⁻(s', a*) ← two-network target
8

Epsilon-greedy solves the exploration-exploitation problem

If the agent always exploits its current best policy, it never discovers better strategies. \(\varepsilon\)-greedy forces random exploration early in training, decaying as the policy matures.

ε: 1.0 → 0.05 over 500 steps (exp decay, τ = 500)
The full pipeline in one sentence: We take an attacker's session history (finite), encode it as a 104-dim state vector, pass it through two neural networks (online and target) that approximate an infinite-horizon Q-function via the Bellman recursion, choose the action with the highest Q-value, and update the weights using the TD error from a random sample of past experience — doing this continuously, online, with every real attacker request.
